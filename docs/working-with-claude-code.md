# Working with Claude Code on this project

If you're following the same `docs/20260423_Morning_briefing_agent_build_guide.docx` walkthrough, you'll notice this repo has a bunch of stuff the guide never mentions: a `pyproject.toml`, a `.pre-commit-config.yaml`, a `CLAUDE.md`, a `.claude/` folder, and a test suite that's bigger than the actual agent code. None of that is required to get a working morning briefing agent — the guide's approach works fine on its own. This doc explains what that extra scaffolding is, why it's there, and how it shaped the way I worked with Claude Code (Anthropic's agentic coding CLI) to build this.

If you haven't used an AI coding agent like Claude Code before: it's a command-line tool you chat with in plain English, and it can read your files, edit them, and run commands on your machine on your behalf (with your approval). Everything below is about *how* to get good results out of that kind of tool, not just this specific project.

## The dev tools, explained for first-timers

You don't need any of this to follow the build guide. It exists because I wanted the agent's behavior to be checked automatically instead of by eyeballing it every time.

- **`uv`** — a Python package and environment manager. Normally, working in Python means manually creating a "virtual environment" and remembering `pip install` commands for every dependency. `uv` does all of that from one config file (`pyproject.toml`) and one command (`uv sync --all-groups`). Every command in this project is run through it (`uv run pytest`, `uv run mypy`, etc.) instead of calling `python` or `pytest` directly, so it always uses the exact right versions of everything. See the commands table in `README.md` for the full list.
- **`ruff`** — a linter and formatter. A *linter* reads your code without running it and flags things that are likely bugs or bad style (an unused variable, a comparison that's always true). A *formatter* automatically rewrites your code to a consistent style (spacing, quote style) so nobody has to argue about it. `ruff` does both.
- **`mypy` (in "strict" mode)** — a type checker. Python doesn't normally require you to declare what type of data a function expects or returns, which means a whole category of bugs ("I passed a string where a number was expected") only show up when the code actually runs. `mypy` reads type annotations in the code and catches those mismatches before you ever run it. "Strict mode" means it also flags places where a type annotation is missing entirely, rather than only checking the ones that exist.
- **`pytest`, plus a 90% coverage gate** — the automated test suite. Every test file under `tests/` calls into the real code and checks the result against what's expected, without any human watching. "Coverage" means what fraction of the actual code got run by at least one test; the "gate" means the test run is considered failed if that fraction drops below 90%. This is meant to make it hard to add code that nothing ever checks.
- **`pre-commit`** — glue that runs `ruff`, `mypy`, and a handful of hygiene checks automatically every time you try to make a commit, so a mistake never even makes it into the project's history. Installing it takes one command (`uv run pre-commit install`) and it then runs itself from then on. It has one sharp edge worth knowing about — see the "Pre-commit" section in `CLAUDE.md` for the specific gotcha (it also enforces the commit-message format below, but only if installed the right way).
- **`commitizen` / Conventional Commits** — this project's git commit messages all follow a `type(scope): subject` pattern (e.g. `fix(calendar): exclude cancelled events`). This isn't just a style preference — `commitizen` actually rejects a commit message that doesn't match the format, via the same `pre-commit` mechanism above. The point is that anyone (or any tool) can scan the commit log and immediately tell what kind of change each commit was, without reading the diff.

## Why test-first (TDD) matters here specifically

The build guide's suggested `agent.py` mixes three things in one file: logging into Google/Slack, making the live API calls, and wiring everything into the agent. That's fine for a one-off script, but it means you can't test any of it without actually hitting Gmail, Calendar, and Slack over the network every time — slow, and it risks touching your real inbox.

This repo splits each data source into three layers instead (see `CLAUDE.md`'s "Source layout rationale" for the details) specifically so the middle layer can be tested against a *fake* client instead of a real one. A fake is a small stand-in object that returns canned data shaped like a real API response, without making any network call. `tests/conftest.py` defines `FakeGmailService`, `FakeCalendarService`, and `FakeSlackClient` for exactly this — look there if you want to see what a fake actually looks like in code.

"Test-first" means: before changing how something behaves, write a test that describes the new behavior, watch it fail (because the behavior doesn't exist yet), and only then write the code that makes it pass. The payoff is that every behavior in this repo has a test proving it works, and that test keeps proving it on every future change — including changes an AI agent makes without you reading every line it wrote.

## Prompting discipline: how I directed Claude Code

The most important habit wasn't any single clever prompt — it was moving anything I'd otherwise repeat in every conversation into a file the agent reads automatically.

- **`CLAUDE.md` as durable instructions.** Rather than re-explaining "use `uv run`, not bare `python`" or "write a test before touching `src/`" in every chat, those rules live in this repo's own `CLAUDE.md`, which Claude Code reads at the start of every session. Open that file — it's the actual instruction set this project was built under, not a summary of one.
- **Small commits scoped to one behavior.** Look at the commit log (`git log --oneline`): `3ddd7b4` fixes exactly one bug (cancelled calendar events weren't filtered), `f1a755c` fixes exactly one other bug (Slack messages lost their sender), and `5e02e74` is a docs-only commit that just records a design investigation that had already happened, with no code change at all. Keeping each commit to one thing makes it possible to review, revert, or explain any single change in isolation — including the reasoning behind it, which future-you (or a classmate, or an AI agent months from now) will want.
- **Encoding a preference as an enforced rule, not a remembered one.** I didn't want any commit in this repo to credit Claude or Anthropic as a co-author. Instead of just asking the agent not to (and hoping it remembers, forever, in every future session), `.claude/hooks/block-claude-coauthor.py` is a small script that Claude Code itself runs automatically before every `git commit`, and it refuses to let the commit through if the message mentions "Claude" or "Anthropic". A *hook*, in Claude Code's vocabulary, is exactly this: a script the tool runs automatically around specific actions, regardless of what any given conversation asked for. Check `.claude/settings.json` to see how it's wired up.

If you're setting up your own project this way, the general pattern worth stealing is: **put anything durable in a file the agent reads, not in chat.** A preference stated only in conversation has to be restated every session (or it silently stops applying); a preference written into `CLAUDE.md` or a hook is enforced automatically from then on. Beyond that: keep commits small and scoped to one behavior, and let the test suite — not a read-through or a vibe check — be what decides whether a change actually works.
